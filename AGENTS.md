# AGENTS.md

## 프로젝트 개요

이 프로젝트는 **"벚꽃살리기"** 라는 이름의 대학 대항 시즌형 웹 서비스이다.

핵심 컨셉은 시험기간 대학생들이 공부에 집중해야 하지만, 실제로는 벚꽃, 학교 경쟁, 친구 공유 같은 요소에 쉽게 흔들린다는 점을 활용하는 것이다.  
사용자는 자신의 학교 계정으로 로그인한 뒤, 벚꽃 붙이기 미니게임에 참여하여 자기 학교의 벚꽃 수와 개화율을 올릴 수 있다.  
또한 랭킹 화면에서 다른 학교를 확인하고, 하루 1회 흔들기 기능을 통해 다른 학교의 벚꽃 수를 줄일 수도 있다.

이 서비스는 생산성 서비스가 아니라, **시험기간의 집중력을 유쾌하게 방해하는 시즌형 참여 서비스**를 목표로 한다.

---

## 프로젝트 목표

이 프로젝트의 핵심 목표는 다음과 같다.

1. 사용자가 짧고 반복적으로 참여할 수 있는 웹 서비스를 만든다.
2. 대학생 타겟에 맞는 학교 소속감, 경쟁심, 시즌성(벚꽃), 시험기간 감성을 살린다.
3. 내 학교를 성장시키고, 다른 학교를 방해하고, 랭킹을 확인하고, 재방문하게 만드는 루프를 만든다.
4. 모바일과 PC에서 모두 동작하는 반응형 웹으로 구현한다.
5. 대회용 프로젝트이므로 **빠른 구현과 명확한 화면 흐름**을 우선한다.

---

## 프로젝트 톤앤매너

이 서비스는 전체적으로 다음과 같은 분위기를 가진다.

- 가볍고 유쾌함
- 시험기간 대학생 밈 감성
- 학교 대항전 느낌
- 봄/벚꽃 시즌성
- 너무 진지하거나 무거운 UX는 지양
- 게임처럼 짧고 직관적인 참여 구조

즉, 사용자가 "잠깐만 해야지" 하고 들어왔다가 여러 번 참여하게 되는 흐름을 목표로 한다.

## 핵심 사용자

주 타겟은 **시험기간의 대학생**이다.

특징:

- 친구가 보낸 링크나 공유에 쉽게 유입될 수 있음
- 모바일로 빠르게 접속할 가능성이 높음
- 짧고 직관적인 미니게임과 랭킹 확인에 흥미를 느낄 수 있음

## 현재 프로젝트 구조에 대한 이해

이 프로젝트는 계정 기반 서비스이다.  
사용자는 회원가입 시 자신의 소속 대학을 선택하고, 이후 로그인하여 자기 학교 상태를 확인할 수 있다.

서비스 핵심 루프는 다음과 같다.

1. 초기화면 진입
2. 로그인 / 회원가입
3. 메인 화면에서 내 학교 상태 확인
4. 벚꽃 붙이기 미니게임 참여
5. 획득 점수만큼 학교 벚꽃 수 / 진행도 / 레벨 반영
6. 랭킹 및 다른 학교 상태 확인
7. 다른 학교 흔들기
8. 재방문 및 반복 참여

## 확정된 주요 기능

### 1. 초기화면
- 배경에 벚꽃나무 이미지 등 시즌성 비주얼 사용
- 중앙에 프로젝트명 **"벚꽃살리기"** 표시
- 사용자가 아무 곳이나 클릭하면 로그인 화면으로 이동

### 2. 로그인 화면
- 아이디 입력
- 비밀번호 입력
- 로그인 버튼
- 회원가입 버튼
- 존재하지 않는 계정 정보로 로그인 시 에러 문구 출력

### 3. 회원가입 화면
- 아이디 입력
- 비밀번호 입력
- 소속 대학 선택
- 완료 시 회원가입 성공
- 완료 후 로그인 화면으로 이동

### 4. 메인 화면
- 중앙에 내 학교의 현재 나무 이미지 표시
- 상단 좌측: 내 학교 현재 전체 순위
- 상단 중앙: 현재 레벨, 다음 레벨까지 진행도
- 상단 우측: 메뉴 버튼
- 하단: 
  - 벚꽃 붙이기 버튼
  - 방해하러가기 버튼

### 5. 벚꽃 붙이기 미니게임
- 화면 상단에서 벚꽃과 벌레가 떨어짐
- 벚꽃 클릭 시 점수 획득
- 벌레 클릭 시 감점
- 게임 종료 후 즉시 메인 화면으로 복귀
- 메인 화면에서 `+점수` 형태로 획득 점수를 시각적으로 보여줌
- 점수는 학교 진행도/벚꽃 수에 반영됨

### 6. 메뉴
메뉴 항목:
- 우리학교 벚꽃 현황
- 모아보기
- 실시간 댓글 커뮤니티
- 로그아웃

동작:
- 우리학교 벚꽃 현황 → 메인 화면
- 모아보기 → 랭킹 화면
- 실시간 댓글 커뮤니티 → 별도 커뮤니티 화면(구현 범위는 미정)
- 로그아웃 → 초기화면으로 이동

### 7. 랭킹 화면
- 기본 정렬은 랭킹순
- 가나다순 보기 가능
- 학교 리스트 제공
- 특정 학교 클릭 시 해당 학교 상세 화면으로 이동

### 8. 다른 학교 상세 화면
- 상단 정보는 메인 화면과 유사하지만, 선택한 다른 학교 기준으로 보여줌
- 해당 학교의 랭킹, 개화율, 레벨/진행도 표시
- 중앙에 해당 학교 나무 화면 표시
- 하단에 흔들기 버튼, 돌아가기 버튼 존재

### 9. 흔들기 기능
- 다른 학교에만 사용 가능
- 계정당 하루 1회 제한
- 상대 학교 총 벚꽃 수 감소
- 사용 후 해당 학교 진행도/개화율/랭킹에 반영

## 현재 임의 기본값 (사용자가 따로 수정하기 전까지 유지)

아래 값들은 아직 최종 확정이 아닌 **임시 기본값**이다.  
사용자가 나중에 수정하기 전까지 이 값을 기준으로 개발/설계를 진행한다.

### 미니게임 규칙
- 게임 시간: 15초
- 벚꽃 클릭: +1점
- 벌레 클릭: -2점
- 게임 종료 후 즉시 메인 화면 복귀
- 획득 점수는 `+10` 같은 형태로 메인 화면 진행도 근처에 표시

### 진행도 및 레벨 계산
- 총 벚꽃 수 누적 기준
- 100점 = 진행도 1%
- 총 5레벨
  - Lv1: 0~19%
  - Lv2: 20~39%
  - Lv3: 40~59%
  - Lv4: 60~79%
  - Lv5: 80~100%
- 진행도 100% 시 만개 상태

### 흔들기 규칙
- 계정당 하루 1회
- 다른 학교만 흔들 수 있음
- 흔들기 1회당 상대 학교 총 벚꽃 수 -30점

### 랭킹 규칙
- 총 벚꽃 수 높은 순으로 랭킹 계산
- 동점이면 먼저 도달한 학교 우선

### 커뮤니티 범위
- MVP에서는 실시간 채팅 대신 간단한 댓글/한줄 응원판 수준으로 가정
- 우선순위는 낮음

### 공유 기능
- 기본은 링크 복사
- 공유용 문구 제공
- 결과 이미지 카드 생성은 보류

### 학교 비주얼
- 학교 로고/마스코트는 저작권/사용권 문제 때문에 기본적으로 사용하지 않음
- 우선은 공통 나무 이미지 + 학교명 텍스트 기반으로 진행
- 학교별 서로 다른 비주얼은 추후 확장 가능

## UI/UX 구현 원칙

1. **모바일 우선**
   - 모바일에서 먼저 보기 좋은 구조로 만든다.
   - 데스크탑에서는 레이아웃을 넓히는 방식으로 대응한다.

2. **짧고 직관적인 흐름**
   - 사용자가 복잡하게 생각하지 않도록 한다.
   - 게임 시작, 결과 반영, 랭킹 확인 흐름이 빠르게 이어져야 한다.

3. **화면 수를 너무 늘리지 않기**
   - 가능한 한 명확한 페이지 구조로 유지한다.

4. **시즌감 있는 비주얼**
   - 벚꽃, 봄, 나무 감성을 살리되 과한 장식보다는 가독성을 유지한다.

5. **게임형 인터랙션**
   - 클릭/감점/랭킹/방해 기능이 명확하게 느껴져야 한다.

- schoolName

프론트는 우선 더미 데이터로 화면부터 구현해도 된다.

중요한 점:

- 나중에 API 연결이 쉽도록 데이터 구조를 단순하게 유지할 것

## 코드 작성 원칙

- 복잡한 추상화보다 명확한 구조를 우선한다.
- 초반에는 빠르게 화면을 완성하는 것이 중요하다.
- 기능이 확정되지 않은 영역은 과도하게 일반화하지 않는다.
- 확장 가능성은 열어두되 MVP 범위를 넘는 구현은 지양한다.
- 타입은 가능한 명확하게 작성한다.
- 주석은 꼭 필요한 부분에만 작성한다.

---

## 현재 절대 잊지 말아야 할 점

1. 이 프로젝트는 **게임형 참여 서비스**이다.
2. 핵심은 **내 학교 성장 + 다른 학교 방해 + 랭킹 확인 + 반복 참여**이다.
3. 미정 세부 규칙은 현재 문서의 **임의 기본값**을 따른다.
4. 학교 로고/마스코트는 기본적으로 사용하지 않는다.
5. 우선은 화면 흐름과 기본 인터랙션 구현이 더 중요하다.
6. 커뮤니티는 후순위다.
7. MVP 범위를 넘는 기능은 사용자가 따로 지시하지 않는 한 먼저 추가하지 않는다.

## 에이전트에게 바라는 작업 방식

이 프로젝트를 도와주는 코딩 에이전트는 아래 기준으로 행동한다.

- 먼저 현재 파일 구조를 파악한다.
- MVP 범위를 기준으로 작업한다.
- 화면 단위로 구현한다.
- 사용자가 별도로 요청하지 않으면 과도한 기능을 추가하지 않는다.
- 미정 세부사항은 본 문서의 임의 기본값을 따른다.
- 더미 데이터 기반 화면 구현 후 실제 데이터 연결 구조를 고려한다.
- 모바일 우선 반응형 UI를 유지한다.
